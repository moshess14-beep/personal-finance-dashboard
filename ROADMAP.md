# Roadmap — מעקב הון אישי

מסמך תכנון חי. מטרתו לשמור הקשר בין סשנים/שיחות פיתוח, כדי שכל המשך פיתוח (גם בסשן חדש) יבנה על ההחלטות הקודמות ולא יסתור אותן.

## עקרון-על: שתי שכבות נתונים נפרדות

האפליקציה בנויה משתי שכבות עצמאיות, בקצב עדכון שונה:

| שכבה | מייצגת | קצב עדכון | Snapshot |
|---|---|---|---|
| **מאזן** (Net Worth) | מה יש לי עכשיו — נכסים והתחייבויות | נכסים/קופות: רבעוני או לפי צורך. הלוואות/עו"ש: חודשי | נוצר בכל "סגירת חודש" (V2) |
| **תזרים חודשי** (Cash Flow) | כמה נכנס/יוצא כל חודש — הכנסות מול הוצאות/חיסכון קבועים | חודשי | נוצר בכל "סגירת חודש" (V2) |

הן **לא** אותו דבר ולא מתערבבות: אפשר שהמאזן יגדל בזמן שהתזרים החודשי שלילי (למשל עליית ערך נדל"ן), ולהפך. **שתיהן ייסגרות יחד באותה פעולת "סגירת חודש"** (V2) — לא כל שדה חייב להשתנות בפועל בכל חודש, אבל ה-Snapshot נוצר מדי חודש בכל מקרה.

## מצב נוכחי (V1 + V1.2 + V1.3 — הושלם ופרוס)

**V1**: מסכי נכסים/התחייבויות (CRUD מלא), מסך חיסכון חודשי (`/savings`), Dashboard עם שווי נקי/KPI/דונאטים/מד נכסים-התחייבויות, גיבוי/שחזור JSON, Dark Mode, GitHub Pages + תיקון ניתוב SPA.

**V1.2** — הזזת האפליקציה מ"מצב נוכחי בלבד" לכיוון מעקב לאורך זמן:
- **תאריכי עדכון**: לכל נכס/התחייבות/רכיב חיסכון/מקור הכנסה יש `updatedAt`, מוצג בכרטיס עצמו ("עודכן היום" / "עודכן לפני X ימים" / תאריך מלא), ובדשבורד כ"עודכן לאחרונה" כללי (`selectLastUpdatedAt` — המקסימום מכל הרשומות).
- **היסטוריה ידנית** (`/history`) — נקודות זמן שהמשתמש מזין ידנית: תאריך, סך נכסים, סך התחייבויות, שווי נקי מחושב אוטומטית, הערה. CRUD מלא. **זו לא עדיין "סגירת חודש" רשמית** — רק דרך פשוטה להזין נקודות עבר/הווה כדי שיהיה על מה לבסס גרף.
- **גרף התקדמות** בדשבורד — קו שווי נקי לאורך זמן, מבוסס `historyPoints` (חייב 2+ נקודות, אחרת empty state מסביר).
- **הכנסות חודשיות — גרסה בסיסית** (`/income`) — רשימת מקורות הכנסה קבועים (עבודה, עסק, שכירות, קצבאות, שוק ההון, אחר), אותו דפוס CRUD כמו חיסכון חודשי. כרטיס דשבורד עם סך ההכנסות + רמז לקטגוריה הדומיננטית.
- מיגרציה: כל השדות החדשים הם תוספות (arrays ריקים כברירת מחדל) — לא נדרש `migrate` שובר, ה-merge הדיפולטיבי של zustand persist מטפל בזה.

**V1.3** — סבב ליטוש מוצר, לפני V2: להפוך את האפליקציה למשהו שאפשר להתחיל להשתמש בו באופן קבוע (בלי AI/תחזיות/סימולציות/ריבית דריבית/לוחות סילוקין/יעדים — נשארים מפורשות מחוץ לסבב הזה).
- **דשבורד "חי" יותר**: אזהרת "עברו X ימים מאז העדכון האחרון" (`NetWorthHero`, מוצג מ-30 יום ומעלה מ-`selectLastUpdatedAt`).
- **ציר זמן** (`/timeline`) — מסך חדש, בסיס לכל ההיסטוריה בעתיד: יומן פעילות כרונולוגי (`activityLog[]` בסטור) שנרשם אוטומטית בכל הוספה/עדכון/מחיקה של נכס, התחייבות, מקור הכנסה, ובכל הוספת נקודת היסטוריה — מקובץ לפי יום, עם אייקון לפי סוג ותג פעולה (נוסף/עודכן/נמחק).
- **KPI חדש בדשבורד**: גידול חודשי בהון (`selectMonthlyNetWorthGrowth`) — מנורמל מהפרש בין שתי נקודות ההיסטוריה האחרונות (דורש פער אמיתי של 14+ יום ביניהן כדי לא לקפוץ למספר קיצוני), או "אין מספיק נתונים" כשאין מספיק נקודות.
- **UX**: הודעות הצלחה זמניות (`useTransientMessage` + `SuccessMessage`) בכל פעולת הוספה/עריכה/מחיקה בכל מסכי ה-CRUD (נכסים/התחייבויות/חיסכון/הכנסות/היסטוריה).
- **גרף ההתקדמות** בדשבורד: עיצוב ציר תאריכים מסתגל — פורמט קצר כשכל הנקודות באותה שנה, כולל שנה כשההיסטוריה חוצה שנים.
- **תיקוני באגים שנמצאו בסבב זה**:
  1. גיבוי JSON (`BackupControls`) לא כלל את `activityLog` החדש בייצוא — חזרה על אותה טעות מ-V1.2 (אז היה חסר `incomeSources`/`historyPoints`). תוקן משורש הבעיה: נוסף `BACKUP_DATA_KEYS` כמקור אמת יחיד בסטור, וה-export נגזר ממנו אוטומטית — כדי שהוספת שדה נתונים עתידי לא תדרוש לזכור לגעת גם בקוד הייצוא בנפרד.
  2. תפריט הניווט (`NavBar`) — הוספת פריט שביעי (ציר זמן) עם תוויות גלויות מ-`sm:` (640px) גרמה ל-overflow אופקי שקט (עם `overflow-x-auto` בלי scrollbar נראה) בכל הטווח 640–1023px. תוקן ע"י דחיית סף הצגת התוויות ל-`xl:` (1280px), עם אימות Playwright שבודק `scrollWidth` מול `clientWidth` בכל רוחב.
- מיגרציה: `activityLog` הוא שדה תוסף בלבד (array ריק כברירת מחדל) — לא נדרש `migrate`, אומת עם דאטה בסגנון V1.2 ישן.

**מודל הנתונים הנוכחי** (`useFinanceStore`): `assets[]`, `liabilities[]`, `savingsComponents[]`, `incomeSources[]` (כולם עם `updatedAt` לכל רשומה) + `historyPoints[]` (נקודות ידניות: `date, totalAssets, totalLiabilities, note`) + `netWorthHistory[]` (רישום יומי אוטומטי קליל למגמת ה-Hero, לא Snapshot רשמי) + `activityLog[]` (יומן פעילות אוטומטי: `entityType, action, summary, timestamp`, מבוסס `BACKUP_DATA_KEYS`).

## V2 — "סגירת חודש" (הבא בתור, מוגדר מחדש)

**הרעיון**: פעולה חודשית מסודרת אחת שיוצרת Snapshot אוטומטי — לא רק המשתמש מזין ידנית נקודה (כמו ב-`/history` היום), אלא האפליקציה מנחה אותו דרך עדכון מסודר וסוגרת את החודש בעצמה.

- נכסים וקופות (נדל"ן, פנסיה, קרן השתלמות) **לא** חייבים עדכון בכל סגירת חודש — עדכון בפועל נשאר רבעוני או לפי צורך; בסגירת חודש שבה לא השתנו, פשוט נשארים כמו שהיו.
- הכנסות, הלוואות/משכנתא, ועו"ש/מזומן — אלה השדות שבאמת משתנים כל חודש, ו"סגירת חודש" מתמקדת בעדכון שלהם.
- בסיום, האפליקציה יוצרת Snapshot אחד לחודש (upsert לפי חודש, כמו `recordNetWorthSnapshot` הקיים) שכולל את כל התמונה — גם צד המאזן וגם צד התזרים.
- **שאלה פתוחה**: מה קורה ל-`historyPoints` הידניים (V1.2) ברגע שיש Snapshots רשמיים? כנראה שהם נשארים כאמצעי לגיבוי נתוני עבר (למשל להזין נקודות משנים קודמות שאין להן Snapshot רשמי), בעוד שגרף ההתקדמות עצמו יעדיף Snapshots רשמיים כשקיימים. להכריע בזמן המימוש.
- גרף שווי נקי אמיתי לאורך זמן (מבוסס Snapshots של "סגירת חודש", לא `netWorthHistory` הזמני ולא בהכרח `historyPoints` הידניים).

## V3 — שכבת "תזרים חודשי" מלאה

**הרעיון**: המשתמש מגדיר פעם אחת רשימת רכיבים קבועים (הכנסות + הוצאות/חיסכון), ובכל "סגירת חודש" (V2) רק מעדכן סכומים שהשתנו.

### מבנה נתונים מתוכנן

```
CashFlowComponent {
  id
  direction: 'income' | 'outflow'
  category: string        // ראו רשימות קטגוריות למטה, תלוי direction
  name: string             // "משכורת משה", "משכנתא"
  currentAmount: number    // "חי" — נערך במקום, נשאר קבוע מחודש לחודש עד שמשנים אותו
  note?: string
  updatedAt
}

// קטגוריות הכנסה: work | business | rental | benefits | capitalMarket | other  (זהה ל-INCOME_CATEGORIES הקיים מ-V1.2)
// קטגוריות הוצאה/חיסכון: mortgage | loan | pensionContribution | kerenHishtalmut | childSavings | additionalDeposit
```

### שאלות פתוחות שצריך להכריע בזמן המימוש

1. מסך `/savings` הקיים (V1) חופף באופן משמעותי לצד ה"הוצאה/חיסכון" של התזרים החודשי. **המלצה**: למזג לתוך מסך "תזרים חודשי" אחד (הכנסות מול הוצאות/חיסכון), במקום מסכים נפרדים.
2. מסך `/income` הבסיסי (V1.2) הוא כבר בדיוק חצי מ-`CashFlowComponent` (direction='income'). כנראה שב-V3 הוא פשוט הופך לצד ההכנסה של אותו מסך, במקום להיבנות מחדש.
3. `MonthlyCashFlowSnapshot` נפרד (כפי שתוכנן קודם) לעומת שילוב בתוך ה-Snapshot היחיד של "סגירת חודש" (V2) — כנראה **השני**: Snapshot חודשי אחד שמכיל גם מאזן וגם תזרים, לא שני מנגנוני Snapshot נפרדים. לעדכן את התכנון כשמגיעים לשם.

## V4 — סטטיסטיקות

מסך שמחשב על סמך נתונים קיימים: שיעור צמיחת הון, שיעור חיסכון מתוך הכנסה (אפשרי כבר מ-V1.2 בזכות `/income` ו-`/savings`), יחס התחייבויות/נכסים, % מכל קטגוריה, הנכס/ההתחייבות הגדולים ביותר.

## V5 — תחזיות

עם V3 מומש, התחזית יכולה להתבסס על תזרים אמיתי (הכנסות בפועל פחות הוצאות בפועל) במקום מספר בודד שהוזן ידנית. תשואה שנתית, אינפלציה, וטווח (5/10/20 שנה) שהמשתמש מגדיר. **חישוב אוטומטי של ירידת קרן הלוואות/משכנתא לפי ריבית ולוח סילוקין עדיין לא נכלל** — עדיין רק סכום שמוזן ידנית בכל עדכון. ייתכן שזה שלב נפרד אחרי V5, לא לפניו.

## מפורש: לא בטווח הקרוב

- ניהול הוצאות מחיה מלא (תקציב, קטגוריות הוצאה שוטפות) — לא מתוכנן עדיין לגרסה ספציפית.
- התחברות משתמשים / ענן — האפליקציה נשארת localStorage + גיבוי JSON ידני בלבד, גם בגרסאות הבאות, אלא אם יוחלט אחרת במפורש.
- OCR / ייבוא אוטומטי מתלושי שכר או דפי בנק.

## רעיונות עתידיים נוספים (עדיין לא מתוכננים לגרסה ספציפית)

- הפרדה בין סוגי צמיחת הון: חיסכון אקטיבי (מהתזרים) מול רווחי הון (עליית שווי נכסים במאזן) מול הכנסה משכירות.
- אחוז מהוצאות המחיה שממומן מהכנסות מנכסים (עצמאות פיננסית) — נגזר משכבת התזרים (V3) ברגע שיש בה קטגוריית "שכירות/שוק ההון" מול הוצאות.
